# Leios homelab load test

Scripts and methodology behind the article **"I ran a Leios node in my homelab"** — a first-hand look at [Ouroboros Leios](https://github.com/input-output-hk/ouroboros-leios), Cardano's scaling protocol, measured on a node running in a home lab on the public **Musashi Dōjō** testnet (June 2026).

Two things are measured here, kept strictly separate:

1. **Network throughput** — measured *passively* by parsing the endorser blocks (EBs) the node receives. This traffic is generated by IOG's own load generators on the testnet, **not** by me.
2. **End-to-end behaviour under my own load** — I submit transactions from the node's own wallet to see how the pipeline batches them, and where *my* setup tops out.

> The 218 TPS peak is the **network's**, driven by IOG's load. My load tests show that **my** transactions enter and get swept into EBs, and where the bottleneck of a casual home setup actually is. Don't conflate the two.

## Measuring throughput (passive)

Each endorser block carries a bitmap of which transactions it includes; a popcount over the bitmap gives the exact tx count. The scripts read `node.log` (JSON logs), dedup by `ebHash`, and reconstruct tx-per-slot from three event types:

- `LeiosFetch.Receive.Block` → `ebBytesSize`
- `LeiosFetch.Receive.BlockTxs` → `bitmaps` (popcount = nº of txs)
- `LeiosFetch.Send.BlockRequest` → `ebSlot`

| Script | What it does |
|---|---|
| [`measure/leios_throughput.py`](measure/leios_throughput.py) | Global + recent-window aggregate (TPS, tx/EB, bytes/EB) |
| [`measure/leios_peak.py`](measure/leios_peak.py) | Sliding-window peak TPS / MB·s⁻¹ per window size + tx/EB distribution |

**Peak throughput curve** (sliding window over ~1M slots of chain history, slotLength = 1 s):

| Window | Peak TPS |
|---|---|
| 10 s | **218 TPS** |
| 30 s | 130 TPS |
| 60 s | 73 TPS |
| 5 min | 32.6 TPS |
| 10 min | 25.5 TPS |

- Largest single EB: **2,184 transactions** · tx/EB distribution p50 = 747, p90 = 2,184.
- For comparison, Cardano mainnet (Praos) has only ever sustained ~12 TPS peak (theoretical ceiling ~18, real average < 1 — [Chainspect](https://chainspect.app/chain/cardano)).

> **Bytes caveat:** `ebBytesSize` (~36 B/tx) is only the size of the tx *references* inside the EB, not the full payload (bodies are diffused separately in Leios). The **tx-count** TPS is the reliable metric; EB byte-throughput underestimates real data (~250 B/tx → 218 TPS ≈ ~55 KB/s).

## Load tests (my own traffic)

Submitting from the pool's wallet to watch end-to-end batching and find the ceiling of a casual setup. Transaction era is **Dijkstra** (`cardano-cli dijkstra transaction build-raw`; the default `build` assembles Conway and the node rejects it on era).

| Script | What it does |
|---|---|
| [`load/consolidate.sh`](load/consolidate.sh) | Merge scattered UTxOs into one (≤200 inputs → 1) — run before a fan-out |
| [`load/load_split.sh`](load/load_split.sh) | Fan-out: split one UTxO into 200 × 5 ADA outputs |
| [`load/load_burst.sh`](load/load_burst.sh) | 200 independent 1-in-1-out txs, pre-signed, fired in a burst |
| [`load/load_sustained.sh`](load/load_sustained.sh) | Sustained submit at N tx/s for M seconds |
| [`load/load_big.sh`](load/load_big.sh) | Larger burst (1,000) |
| [`load/load_limit.sh`](load/load_limit.sh) | Flood: K txs at parallelism P (push the client to its limit) |
| [`load/orchestrate.sh`](load/orchestrate.sh) | consolidate → sustained → consolidate → limit |

### Results

| Scenario | Submit rate | Inclusion | Notes |
|---|---|---|---|
| Burst 200 | — | 1 EB, ~34 s | all wait, a single EB sweeps them together |
| Burst 1,000 | ~82 tx/s | ≥3 EBs, ~56 s | concurrent drain (581 already confirmed by end of burst) |
| Sustained 10 tx/s × 150 s | ~2 tx/s real | mempool near-empty | client-limited; the network drains faster than I inject |
| **Flood** 2,000, parallelism 80 | **557 tx/s** | drains in ~110 s | **1,673 ok / 327 err** |

**The bottleneck is the client, not the node or Leios.** At parallelism 80 the flood fails with `Network.Socket.connect: resource exhausted` (327/2,000) — local socket exhaustion from forking `cardano-cli` per tx, *not* mempool rejection. Meanwhile:

- Mempool peak: **1,639 tx / 375 KB** (cap 500 KB) — never fully filled.
- Node CPU: peak load avg **1.89 / 4 cores (~47%)** — it doesn't flinch.

To actually stress Leios you'd need a persistent submitter (not one `cardano-cli` fork per tx) or IOG's `tx-generator`. The network — Leios plus the testnet's pools — absorbs everything a home node can throw at it.

## Notes & gotchas

- `consolidate.sh` matters: the fan-out assumes one large UTxO. Without consolidating first, funds fragment and the fan-out jams (`tx does not balance` / `AllInputsAreSpent`).
- `calculate-min-fee` returns JSON `{"fee":N}` — parse with `jq -r .fee` and add a buffer.
- `stake-address` / `stake-pool` live under `cardano-cli latest …` (not top-level) in this build.
- Paths in the scripts (`/opt/leios/...`) are from my LXC layout — adjust `CARDANO_NODE_SOCKET_PATH`, key paths and `--testnet-magic` for yours.
